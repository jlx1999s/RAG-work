# 医疗 RAG 四大能力实现详解

本文聚焦本项目的四个核心方向：

1. 双路径检索
2. Chunk 策略
3. 长短期记忆与用户画像
4. 评测闭环

每个方向都回答三个问题：**怎么做、特点是什么、还有什么不足**。

---

## 1. 双路径检索

### 1.1 怎么做

本项目在检索层实现了三种模式：`vector_only / hybrid / graph_only`，并支持自动路由。

- 向量检索：
  - 使用 Milvus 检索文档 chunk；
  - 支持原问题 + 子问题联合检索；
  - 结果会做去重、文本/图表块分流、基础融合。
- 图检索：
  - 使用 LightRAG/Neo4j 获取实体关系相关片段；
  - 支持超时控制、缓存、熔断与恢复机制。
- 融合检索（hybrid）：
  - 并行触发向量与图检索；
  - 使用 RRF（Reciprocal Rank Fusion）做初步融合；
  - 加入医学优先加分（如禁忌、剂量、推荐等级等 chunk role）；
  - 后续做语义重排 + MMR 多样性选择，最后输出融合结果。
- 条件跳过图检索：
  - 当向量结果数量和置信度足够高时，可跳过图检索降低延迟；
  - 同时保留图检索熔断保护，避免图侧异常拖垮整体链路。

关键实现位置：

- `rag-backend/backend/agent/graph/raggraph_node.py`
  - `vector_db_retrieval_node`
  - `graph_db_retrieval_node`
  - `hybrid_retrieval_node`
  - `_merge_retrieved_docs`
  - `_mmr_select_docs`

### 1.2 特点

- 检索策略不是单一路径，能兼顾语义召回和关系推理。
- 融合阶段有明确统计输出（`retrieval_fusion_stats`），可观测性较好。
- 具备缓存、超时、熔断、条件降级等工程化保护手段。

### 1.3 不足

- 自动路由仍以规则+阈值为主，对边界问题鲁棒性有限。
- 图检索质量强依赖图谱构建质量，实体抽取错误会被放大。
- RRF/MMR 参数主要靠经验调参，缺少系统化参数搜索与数据驱动优化。
- 缺少“查询意图分层评测”（事实查询、关系查询、复杂约束查询）来精准证明每条路径收益。

### 1.4 优先改进建议

- 建立路由分类器离线训练集，替代纯规则阈值。
- 对图谱构建加入质量诊断（孤点率、错误关系抽样）。
- 为 RRF/MMR 建立自动调参脚本和固定回归基线。

## 2. Chunk 策略

### 2.1 怎么做

项目支持多种分块策略：`character / semantic / recursive / markdown_header / medical_hybrid`。

- 医疗场景核心使用 `medical_hybrid`：
  - 先按 Markdown 章节拆分；
  - 对过短章节进行合并；
  - 再用递归分块切成检索单元；
  - 为每个 chunk 打医疗元数据标签。
- 元数据增强包括：
  - `section_title`
  - `chunk_role`（contraindication / dosage / recommendation 等）
  - `evidence_level`
  - `is_key_clause`
- 检索融合阶段会使用这些元数据做权重加成（关键条款更容易进入最终答案）。

关键实现位置：

- `rag-backend/backend/rag/chunks/chunks.py`
- `rag-backend/backend/rag/chunks/models.py`

### 2.2 特点

- 分块不只是“切长度”，而是加入了医疗语义角色信息。
- 分块结果会直接影响检索权重，形成“分块-检索-生成”联动。
- 支持普通文档和医疗文档双场景策略切换。

### 2.3 不足

- `chunk_role` 识别主要依赖关键词规则，覆盖面和准确率有限。
- 章节依赖 Markdown 结构，OCR 或格式噪声会影响分块质量。
- 目前缺少 chunk 级别标注集，无法精确量化“哪类 chunk 策略更优”。
- 图表、表格、药品说明书等特殊结构的 chunk 策略仍可进一步细化。

### 2.4 优先改进建议

- 为高价值字段（禁忌、剂量、证据等级）构建弱监督标注集。
- 增加 chunk 质量审计指标（覆盖率、重复率、断句合理性）。
- 对图表/表格引入专门的结构化切块和召回链路。

## 3. 长短期记忆与用户画像

### 3.1 怎么做

项目将“短期会话记忆”和“长期用户记忆”分开处理。

- 短期记忆（会话内）：
  - 保留最近若干轮 user/assistant 对话；
  - 历史过长时会生成对话摘要，作为当前会话背景信息；
  - 主要用于当前会话连续性。
- 长期记忆（跨会话）：
  - 从用户消息抽取画像候选（语言偏好、风格、姓名、年龄、慢病、过敏等）；
  - 存储 profile 与 event 两类记忆；
  - 支持 TTL、去重、重要性评分、相关性排序、主题多样性、token 预算控制；
  - 支持 Redis 缓存，部分场景支持记忆向量召回增强。
- 回答阶段：
  - 根据当前 query 拉取相关记忆上下文；
  - 通过 system prompt 注入，影响回答风格和内容连续性。

关键实现位置：

- `rag-backend/backend/service/memory.py`
- `rag-backend/backend/service/chat.py`（会话摘要 + 注入流程）

### 3.2 特点

- 记忆结构化程度较高，区分稳定画像和动态事件。
- 有较完整的工程控制：缓存、去重、相关性排序、预算控制、TTL。
- 已考虑记忆合规（部分字段白名单、合规开关）与性能开销平衡。

### 3.3 不足

- 用户画像抽取主要靠规则/正则，复杂表达下召回与准确率仍有提升空间。
- “记忆是否真正提升回答质量”的量化证据还不够系统化。
- 多用户、多知识库并发场景下的记忆一致性与冲突策略仍可加强。
- 医疗敏感信息脱敏与最小化存储策略还需要更明确的产品化规范。

### 3.4 优先改进建议

- 引入轻量结构化抽取模型替代部分规则抽取逻辑。
- 建立“记忆开/关”A/B 评测，量化对多轮一致性提升。
- 补充敏感字段分级存储和可配置脱敏策略。

## 4. 评测闭环

### 4.1 怎么做

项目具备离线脚本 + 在线 API 双评测入口。

- 离线评测：
  - 使用 `eval_ragas.py` 读取 JSONL 数据集；
  - 输出质量指标与样本级结果。
- 在线评测：
  - 提供 `/rag/evaluate` 与 `/rag/evaluate-async`；
  - 支持评测任务状态查询、历史保存、数据集保存/读取；
  - 输出质量、性能、稳定性、检索统计等结构化报告。
- 指标维度：
  - 质量：`context_precision / context_recall / answer_relevancy / faithfulness`
  - 性能：`avg/p50/p95/p99 latency`, `throughput`
  - 稳定性：`error_rate`, `empty_answer_rate`
  - 检索：来源分布、融合统计（vector_docs、graph_docs、merged_docs 等）

关键实现位置：

- `rag-backend/backend/tests/eval_ragas.py`
- `rag-backend/backend/api/rag.py`
- `rag-frontend/src/views/Evaluation.vue`

### 4.2 特点

- 从“可运行”走向“可证明”：指标体系较完整，支持历史回看。
- 评测结果可下钻到 item 级别，便于定位问题样本。
- 具备基础工程化能力（异步任务、历史记录、配置化评测参数）。

### 4.3 不足

- 医疗评测集规模偏小，覆盖疾病类型与问法多样性不足。
- 当前仍以通用 RAGAS 指标为主，缺少医疗特有安全指标。
- 离线评测和线上真实用户反馈尚未形成统一闭环。
- 数据集与历史存储策略仍有进一步规范空间（多租户隔离、版本管理）。

### 4.4 优先改进建议

- 扩充数据集并分层（事实问答、关系推理、高风险问答、拒答场景）。
- 新增医疗安全指标（危险建议率、急症分流触发准确率、无依据断言率）。
- 建立评测版本管理和固定基线报告模板。

---

## 总结（给面试官的一句话）

本项目的价值不只是“做了一个医疗问答机器人”，而是围绕四个核心工程能力（双路径检索、Chunk、记忆画像、评测）建立了可迭代、可解释、可控风险的医疗 RAG 实践框架。
